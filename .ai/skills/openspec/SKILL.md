# OpenSpec

## When To Use

Use when the task should be expressed as an OpenSpec proposal, design, task plan, implementation workflow, or archive step.

## Core Rules

- Match the existing OpenSpec workflow already used in the repo.
- Pick the narrowest OpenSpec action that fits the request.
- Keep planning artifacts aligned with the actual implementation scope.

## Preferred Patterns

- Explore before proposing when the problem is still fuzzy.
- Propose when the change needs formal scope and tasks.
- Apply when tasks are implementation-ready.
- Archive when implementation is complete and the change is ready to close.

## Anti-Patterns

- Starting implementation when the change is still ambiguous.
- Creating extra OpenSpec artifacts that do not help execution.
- Treating archived changes as active planning input without checking date and status.

## Checklist

- Identify whether the user needs explore, propose, apply, or archive behavior.
- Read existing change artifacts before extending them.
- Keep implementation and task tracking in sync.
